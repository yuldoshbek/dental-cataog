# Bitrix24 — Смарт-процесс «Каталог оборудования»

Документ описывает, как создать смарт-процесс в Bitrix24 и подключить его к каталогу.

---

## 1. Что такое смарт-процесс в данном контексте

Смарт-процесс — это база товаров стоматологического оборудования внутри Bitrix24.
Каждая запись = один товар (установка, компрессор, сканер и т.д.).

**Принцип работы:**
```
Менеджер редактирует карточку товара в Bitrix24
        ↓
Сервер каталога запрашивает данные через REST API (кэш 60 сек)
        ↓
Менеджер видит актуальный каталог на сайте
```

Каталог работает в режиме **только чтения** из Bitrix. Создавать и редактировать товары нужно в Bitrix24.

---

## 2. Создание смарт-процесса в Bitrix24

### Шаг 1 — Создать смарт-процесс

1. Зайти в **CRM → Ещё → Смарт-процессы**
2. Нажать **Добавить**
3. Название: `Каталог оборудования`
4. Включить: **Контакты**, **Компании** — по желанию
5. Сохранить

> После создания Bitrix присвоит смарт-процессу **entityTypeId** (число).
> Запишите его — понадобится для настройки.
> Найти: в URL при открытии смарт-процесса (`typeId=XXX`).

### Шаг 2 — Настроить стадии (воронку)

Перейти в **Настройки → Стадии** и создать:

| Название стадии     | Внутренний код (stageId)  | Описание                          |
|---------------------|---------------------------|-----------------------------------|
| Черновик            | `DT{typeId}_1:NEW`        | Товар заполняется, не виден       |
| На проверке         | `DT{typeId}_1:REVIEW`     | Ожидает одобрения                 |
| Готов (внутренний)  | `DT{typeId}_1:READY`      | Виден только менеджерам           |
| Опубликован         | `DT{typeId}_1:SUCCESS`    | Виден в каталоге и клиентах       |
| Архив               | `DT{typeId}_1:FAIL`       | Снят с публикации                 |

> Система автоматически определяет статус по ключевым словам в stageId:
> - `publish` / `success` → `published` (виден в каталоге)
> - `archive` / `fail` → `archived`
> - `ready` → `ready_internal`
> - `review` → `review`
> - всё остальное → `draft`

### Шаг 3 — Добавить пользовательские поля (UF_*)

В настройках смарт-процесса → **Поля** → добавить:

#### Основная информация

| Название поля       | Тип         | Код поля (UF_CRM_{typeId}_*)  | Обязательное |
|---------------------|-------------|-------------------------------|--------------|
| Категория           | Строка      | `CATEGORY`                    | ✅           |
| Бренд               | Строка      | `BRAND`                       | ✅           |
| Модель              | Строка      | `MODEL`                       | ✅           |
| Страна              | Строка      | `COUNTRY`                     | ✅           |

#### Цены

| Название поля       | Тип         | Код поля                      |
|---------------------|-------------|-------------------------------|
| Цена (текст)        | Строка      | `PRICE_LABEL`                 |
| Цена минимальная    | Число       | `PRICE_MIN`                   |
| Цена средняя        | Число       | `PRICE_AVG`                   |
| Цена максимальная   | Число       | `PRICE_MAX`                   |

#### Описания

| Название поля         | Тип            | Код поля             |
|-----------------------|----------------|----------------------|
| Описание менеджера    | Текст (многострочный) | `MANAGER_DESCRIPTION` |
| Описание клиента      | Текст (многострочный) | `CLIENT_DESCRIPTION`  |

#### Характеристики

| Название поля         | Тип     | Код поля        |
|-----------------------|---------|-----------------|
| Характеристики        | Текст   | `SPECS`         |
| Цвета                 | Строка  | `COLORS`        |
| Обивка                | Строка  | `UPHOLSTERY`    |
| Базовая комплектация  | Текст   | `BASE_CONFIG`   |
| Опции                 | Текст   | `OPTIONS`       |

#### Категорийные поля (для отдельных категорий)

| Название поля              | Тип     | Код поля      | Для категории       |
|----------------------------|---------|---------------|---------------------|
| Подходит для установок     | Строка  | `FOR_UNITS`   | Компрессоры         |
| Осушитель                  | Строка  | `DRYER`       | Компрессоры         |
| Чехол                      | Строка  | `COVER`       | Портативное         |
| Тип                        | Строка  | `TYPE`        | Автоклавы, рентгены |
| Количество цилиндров       | Строка  | `CYLINDERS`   | Компрессоры         |
| Габариты                   | Строка  | `DIMENSIONS`  | Любые               |

#### Медиа

| Название поля    | Тип      | Код поля         |
|------------------|----------|------------------|
| Главное фото     | Файл     | `PRIMARY_IMAGE`  |
| Галерея          | Файл (список) | `GALLERY`  |

#### Публикация и идентификаторы

| Название поля          | Тип             | Код поля            |
|------------------------|-----------------|---------------------|
| Slug (ссылка)          | Строка          | `SHARE_SLUG`        |
| Внешний код            | Строка          | `EXTERNAL_CODE`     |
| Активен                | Да/Нет (boolean) | `IS_ACTIVE`        |
| Публиковать в каталоге | Да/Нет          | `PUBLISH_WEB`       |
| Публиковать клиентам   | Да/Нет          | `PUBLISH_SHARE`     |
| Публиковать в Telegram | Да/Нет          | `PUBLISH_TELEGRAM`  |

## 3. Создание Webhook для REST API

1. В Bitrix24: **Настройки → Интеграции → Входящие вебхуки → Добавить**
2. Разрешить методы:
   - `crm.item.list`
   - `crm.item.get`
3. Скопировать URL вида: `https://your-domain.bitrix24.ru/rest/1/XXXXXXXXXXXX/`

---

## 4. Переменные окружения

Добавить в переменные среды сервера (`.env` или Панель управления VPS):

```env
# Включить Bitrix провайдер (по умолчанию 'local' = SQLite)
CATALOG_PROVIDER=bitrix

# URL входящего вебхука из шага выше
BITRIX_WEBHOOK_URL=https://your-domain.bitrix24.ru/rest/1/XXXXXXXXXXXX/

# entityTypeId вашего смарт-процесса (число из URL)
BITRIX_SMART_PROCESS_ENTITY_TYPE_ID=1234

# Кэш в миллисекундах (60000 = 1 минута, рекомендуется)
BITRIX_CACHE_TTL_MS=60000
```

### Маппинг полей (если коды полей отличаются)

По умолчанию система ожидает поля в camelCase формате Bitrix API.
Если ваши поля называются иначе — переопределите через JSON:

```env
BITRIX_FIELD_MAP_JSON={"brand":"ufCrm5Brand","model":"ufCrm5Model","categoryId":"ufCrm5Category"}
```

> **Как узнать точные имена полей?**
> Вызовите `crm.item.list` через браузер (добавьте `/crm.item.list.json?entityTypeId=XXX` к вебхук URL).
> В ответе увидите реальные имена полей (например `ufCrm5Brand`).

### Маппинг категорий (если в Bitrix хранятся другие значения)

```env
BITRIX_CATEGORY_MAP_JSON={"Установки":"units","Компрессоры":"compressors","Автоклавы":"autoclaves","Физиодиспенсеры":"physio","Интраоральные сканеры":"scanners","Рентгены портативные":"xray","Визиографы":"visiographs","Наконечники":"handpieces"}
```

### Маппинг стадий (если нужно переопределить логику)

```env
BITRIX_STAGE_MAP_JSON={"DT1234_1:SUCCESS":"published","DT1234_1:FAIL":"archived"}
```

## 5. Категории — допустимые значения

Поле `CATEGORY` должно содержать одно из значений:

| Значение      | Название               |
|---------------|------------------------|
| `units`       | Установки              |
| `compressors` | Компрессоры            |
| `autoclaves`  | Автоклавы              |
| `physio`      | Физиодиспенсеры        |
| `scanners`    | Интраоральные сканеры  |
| `xray`        | Рентгены портативные   |
| `visiographs` | Визиографы             |
| `handpieces`  | Наконечники            |

Система также распознаёт русские названия категорий через `BITRIX_CATEGORY_MAP_JSON`.

---

## 6. Как работает публикация

Товар **виден в каталоге** если выполнены все три условия:
1. Стадия → `published` (содержит `publish` или `success` в stageId)
2. Поле `IS_ACTIVE` = `Да`
3. Поле `PUBLISH_WEB` = `Да` (или не задано — тогда берётся из стадии)

Товар **виден клиенту (share-ссылка)** если дополнительно:
- Поле `PUBLISH_SHARE` = `Да`

---

## 7. Как заполнять поле SLUG (ссылка клиенту)

Поле `SHARE_SLUG` — уникальный идентификатор для share-ссылки.

Формат: `brand-model` строчными буквами, через дефис, латиница.

Примеры:
- `ajax-aj15-premium`
- `shining-aoralscan3-wireless`
- `kavo-e70`

Ссылка клиенту: `https://ваш-сайт.ru/share/ajax-aj15-premium`

Если поле не заполнено — система использует числовой ID из Bitrix.

## 8. Проверка интеграции

После настройки переменных:

```bash
# Убедитесь, что сервер стартует без ошибок
npm run server

# Проверьте, что API возвращает товары из Bitrix
curl http://localhost:3001/api/products

# Проверьте категории
curl http://localhost:3001/api/categories

# Проверьте шпаргалку
curl http://localhost:3001/api/categories/units/summary
```

Если `CATALOG_PROVIDER=bitrix` не задан — сервер использует **локальную SQLite** базу.

---

## 9. Ограничения режима Bitrix

В режиме `CATALOG_PROVIDER=bitrix`:

- Создание/редактирование/удаление товаров через API **недоступно** (ошибка 409)
- Все изменения делаются **в Bitrix24**
- Данные кэшируются на `BITRIX_CACHE_TTL_MS` мс (по умолчанию 60 секунд)
- Заявки клиентов (`/api/inquiries`) сохраняются в локальной SQLite таблице `external_inquiries`
